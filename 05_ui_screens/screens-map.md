## Общие правила
- O-1 — точка входа, без «Назад»
- O-2 — O-15: линейная цепочка с прогресс-баром (шаг X/14, или X/13 если O-4 пропущен)
- O-1, O-16, O-17 — без прогресс-бара
- Условные экраны: O-4 (ветка похудения), O-7 (только женщинам)
- После онбординга → H-1 (Дашборд)
- Таб-бар: Главная / План / Покупки / Прогресс / Профиль
- Модальные экраны: PH-1 (фото-анализ) — без таб-бара
- Язык: только русский, обращение на «ты». В онбординге строго используется формат унисекс — глаголы с окончанием «(а)» (указал(а), готов(а)). Внутри приложения (дашборд, рецепты, уведомления) обращение должно быть строго персонализировано по полу пользователя, который мы уже знаем из профиля.
- Все заболевания, аллергии и пищевые ограничения учитываются на ВСЕХ статусах включая White
- Первые 3 дня — триал Gold (полный функционал)
- **Frontend-интеграция**: Экраны используют JS (например, `health-analytics.js`, `fasting-state.js`) для объединения в бесшовную систему без серверной части (Zero-Knowledge подход).
- **Статусы** в продукте: Статус White / Статус Black / Статус Gold. Во время ревью Apple активируется `Stealth Review Mode` (через `app-config.js`), который полностью скрывает концепцию статусов и открывает весь функционал приложения.

### Навигационный контракт (Edit Mode)

Когда пользователь нажимает «Изменить» на экране O-16 (Сводка), он переходит на соответствующий экран онбординга с параметром `?from=summary`.

**Поведение в edit mode (`?fromSummary=true`):**
1. Кнопка «← Назад» → заменяется на **«← Отмена»** → возврат на O-16
2. Кнопка «Далее →» → заменяется на **«✓ Сохранить»** → сохранение + возврат на O-16
3. При нажатии «Изменить», O-16 запоминает ID редактируемой секции. При возврате экран автоматически прокручивается к этой секции (anchor scroll через `Scrollable.ensureVisible()`), а не открывается с начала.
4. На экранах без кнопки «Назад» (O-1) — кнопка «← Отмена» инжектится динамически

**Реализация (Flutter):** `OnboardingScaffold` → `isFromSummary`, `O16SummaryScreen` → `_scrollToSection` + `_routeToSection` + `GlobalKey`

- Поле «Телосложение»: выпадающий — Худощавое / Среднее / Крепкое / Спортивное / Есть лишний вес.
- Поле «Обхват талии (см)»: числовое. Подсказка (ℹ️): «Измерь сантиметровой лентой на уровне пупка, стоя, на выдохе.» Чекбокс: «Не знаю / измерю позже».

Автоматические расчёты (не показываются, записываются в профиль):
- ИМТ = вес / (рост/100)²
- Соотношение талия/рост = waist / height
- BMR по Миффлину — Сан Жеору: М = 10×вес + 6.25×рост − 5×возраст + 5; Ж = 10×вес + 6.25×рост − 5×возраст − 161
- Классификация ИМТ: <18.5 недостаточный / 18.5–24.9 нормальный / 25–29.9 избыточный / ≥30 ожирение

> Для этих трёх целей чекбокс **СКРЫТ ПОЛНОСТЬЮ**.

## Глобальные стандарты и механики Onboarding'а (Обновление)

1. **Глобальный Interceptor Автосохранения**:
В файле `onboarding-state.js` внедрена логика перехвата события `click` на фазе `Capture` (capture: true). При нажатии пользователем любой кнопки «Далее», скрипт мгновенно останавливает браузер, извлекает все значения из `input` и `textarea` и сохраняет их в `aidiet_profile` через `window.ejeweeka.saveField`. Это полностью защищает проект от потери данных при быстрой навигации (когда blur event не успевает сработать на мобильном Safari).

2. **Стандарт полей ввода (Flex-shrink Prevention)**:
Все текстовые поля ввода во всем проекте стандартизированы. 
Запрещено использовать кастомные хаки или position: absolute для единиц измерения внутри `input-wrapper` без нативной flex-поддержки:
```css
.input-wrapper { min-height: 52px; flex-shrink: 0; display: flex; align-items: center; }
.input-field { min-height: 48px; flex: 1; border: none; }
```
