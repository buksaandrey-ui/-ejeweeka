## 2. Общая идея

ejeweeka — умный наставник по питанию, сну и физической активности. Приложение работает по принципу «офлайн-первый»: основные данные хранятся на телефоне пользователя, а сервер используется для генерации персональных планов, алгоритмической обработки данных, синхронизации и платежей.

---

## 3. Позиционирование продукта (Mass-Market Wellness)

Массовый Apple-like wellness-продукт: простой, чистый, умный, доброжелательный помощник по питанию, сну и активности, понятный миллионам пользователей. 
Приложение строго соблюдает Guideline 1.4.1 (отказ от медицинской диагностики). Все рекомендации, генерируемые на базе знаний, носят исключительно wellness-характер и фильтруются для исключения любых диагностических заявлений. Никакого искусственного интеллекта в виде слепых генераций — только проверенные нутрициологические алгоритмы, поданные в дружелюбной и заботливой форме.

## 4. Шесть ключевых УТП

1. **Всё в одном плане** — «Что есть, как готовить, что купить, как тренироваться — всё в одном плане под твой бюджет и время»
2. **Оптимизация корзины и времени** — «Умное переиспользование ингредиентов (Cross-Utilization) и адаптация под частоту закупок. Алгоритм минимизирует остатки продуктов в холодильнике и генерирует план заготовок, если ты готовишь раз в неделю.»
3. **Здоровье микробиома** — «Худей через здоровье кишечника. Мы встроим природные пребиотики (пищу для микрофлоры) в каждый твой день, чтобы иммунитет и пищеварение работали как часы»
4. **Витамины, которые работают** — «Железо с кофе — деньги на ветер. Витамин D без жирной еды не усвоится. Мы составим расписание витаминов, которое реально работает»
5. **Учтём все нюансы** — «Веган, халяль, беременность, подагра — учтём все важные для тебя нюансы»
6. **Калории по фото** — «Ужин в ресторане или в гостях? Просто сфотографируй — смарт-алгоритм всё посчитает и пересчитает план»

## 5. Слои приложения (как бутерброд)

Приложение состоит из 4 слоёв, снизу вверх:

### Слой 1 — Телефон пользователя (клиент)
- Что: мобильное приложение iOS/Android
- Хранит локально: имя, аватар, настройки, кеш плана питания
- Подключается к: Apple Health / Google Health Connect (сон, шаги, тренировки, вес)
- Локальная БД: SQLite / Room (Android) / CoreData (iOS)
- Синхронизация: SyncManager отправляет данные на сервер когда есть интернет

### Слой 2 — API Gateway (входная дверь сервера)
- Что: единая точка входа для всех запросов с телефона
- Делает: проверяет JWT-токен, направляет запрос в нужный сервис, ограничивает частоту запросов (rate limiting)
- Технология: Nginx или Traefik

### Слой 3 — Серверные сервисы (микросервисы)

**Auth & Billing Service (Hybrid Monetization)**
- Анонимный вход: сервер генерирует `UUID` при вызове `/init`, выдаёт JWT токен и сохраняет хеш `anonymous_uuid_hash`.
- Унификация (Hybrid Auth): `/init` создаёт **одновременно** старую модель `User` (для совместимости legacy роутеров) и новую модель `AppProfile` + `Entitlement` (source of truth для статусов и биллинга). Мобильное приложение парсит `app_profile_id` и `entitlement_status` для синхронизации.
- **Опциональная привязка email**: пользователь может привязать email (one-time magic code) для синхронизации статуса между устройствами и платформами. Email хранится зашифрованным (`encrypted_email` + `email_hash`).
- **Два канала оплаты**:
  - **iOS/Global**: Apple IAP (auto-renewable subscriptions) через StoreKit. Транзакции валидируются backend'ом.
  - **Web РФ (ejeweeka.ru)**: ЮKassa (СБП, карты РФ). Промокоды доступны ТОЛЬКО на web.
- **Unified Entitlement System**: Единая таблица `entitlements` — source of truth для статуса (white/black/gold). Source: `trial`, `app_store`, `web`, `reviewer_demo`, `manual_support`.
- **Review Demo Mode**: при модерации Apple ревьюер использует тестовый email (`review@ejeweeka.app`, code `000000`) для получения gold-статуса.
- **Запрещено в iOS-приложении**: ссылки на web checkout, ЮKassa, СБП, Telegram-оплата, промокоды, deep links на /subscribe.
- Технологии: Python FastAPI + Apple App Store Server API + ЮKassa API

**Core API (Stateless шлюз)**
- Работает по принципу Zero-Knowledge: принимает JSON-пакет с медицинскими данными, прогоняет через Recommendation Engine и возвращает ответ.
- НЕ сохраняет `plan`, `profile`, `дневники` в серверную БД. Все CRUD-операции происходят ТОЛЬКО в локальной SQLite пользователя.
- Технологии: Python FastAPI

**Recommendation Engine (Ядро AI-аналитики)**
Самый сложный модуль системы, работающий по принципу **многослойного медицинского фильтра**. Генерация плана проходит через 5 этапов алгоритмического процесса.

- **База знаний**: 16 000+ экспертных материалов от 30+ практикующих врачей → транскрипция (Whisper) → суммаризация → чанки → embeddings → семантический поиск (pgvector).
- **5-Стадийная Математическая Модель**:
  1. **Валидация (Cleanse & Audit)**: Сквозной аудит 61 параметра онбординга (биометрия, 11 мед-специализаций, быт, психология). Мгновенная блокировка ингредиентов-аллергенов (P0).
  2. **Баланс (BMR/TDEE)**: Расчет по формуле Миффлина — Сан Жеора с учетом `activity_multiplier`.
  3. **Траектория (Pace)**: Динамический расчет дефицита (7700 ккал/кг жира). Дефицит ограничен 25% от TDEE (кроме aggressive). Жесткие пороги безопасности: 1200 ккал (жен) / 1500 ккал (муж).
  4. **Гарды (Guardrails)**: Авто-повышение белка до 1.2–1.5 г/кг (саркопения 45+), контроль инсулина (СПКЯ/Диабет: ГИ < 50), фильтрация Drug-Food конфликтов, компенсация метаболического стресса при голодании (электролиты).
  5. **Беременность/Лактация Guardrail**: Запрет любого дефицита калорий для беременных (минимум TDEE + 340) и кормящих (минимум 1800 ккал или TDEE - 500).
  6. **Адаптация (Correction Loop)**: Пересчет плана на лету на основе логов (`extra_snacks` > 15% нормы → повышение клетчатки на завтра; `beverages` с алкоголем → инъекция гепатопротекторов и калия).
  6. **Гео-Нутрициология (Geo-Cultural)**: Фильтрация продуктовой корзины на основе страны (например, исключение гречки в Азии, адаптация фокуса на Витамин D в зависимости от УФ-индекса).
  7. **Спортивная Динамика**: Распределение профицита углеводов и белка строго в дни тренировок ("углеводное окно") с сохранением дефицита в дни отдыха.

- **Иерархия Исключений (Conflict Resolution)**:
  1. **БЕЗОПАСНОСТЬ (P0)**: Аллергии и медицинские противопоказания (не обсуждается).
  2. **БЮДЖЕТ (P1)**: Если диета слишком дорога для `budget_level`, AI обязан выбрать бюджетные аналоги (субпродукты, сезонные овощи).
  3. **ВКУС (P2)**: Любимые/нелюбимые продукты.

- **SSOT: Единые математические константы**:
  * **Fat Caloric Density**: 7700 kcal / kg (WHO Standard)
  * **Min Calories**: 1200 kcal (F) / 1500 kcal (M) (Metabolic Safety Floor)
  * **Sarcopenia Protein**: 1.2 - 1.5 g/kg (Gerontology Standard)
  * **Fiber Target**: 25g (F) / 30g (M) (AHA / WHO)

- **RAG-Контекст**: Поисковый запрос включает не только болезни, но и текущие симптомы, диетические ограничения и данные о женском здоровье для максимально точного подбора медицинских советов.
- **Photo Analysis v1.1** (распознавание блюд по фото через Gemini Vision).
- **Персональный HC-чат** (C-1): RAG-поиск по базе знаний врачей.
- **Проверка совместимости**: Автоматический аудит взаимодействий «лекарство-еда» и «витамин-витамин».
- Технологии: Python FastAPI + Gemini API + pgvector + Whisper.

**Archetype Prompt Factory (`archetypes.py`)**
Трёхслойная композиционная система промптов для персонализированной генерации планов. Заменяет монолитный `_build_system_role` в `assembler.py`.

- **Layer 1 — Archetype System Prompt**: Определяет тон, философию и базовые макро-правила. Собирается из 3 компонуемых блоков:
  - *Goal Cluster* (5 кластеров): `DEFICIT` (weight_loss, reduce_cravings), `SURPLUS` (muscle_gain), `BALANCE` (maintenance, improve_energy), `THERAPEUTIC` (health_restrictions, recovery), `AESTHETIC` (skin_hair_nails, age_adaptation)
  - *Gender Modifier* (2): male, female — гормональный профиль, приоритетные микроэлементы
  - *Life Stage Modifier* (3): young (18-39), mature (40-59), senior (60+) — саркопения, кальций, B12
  - *Special Overrides* (2): Pregnancy, Breastfeeding — полное переопределение калорийной стратегии
- **Layer 2 — Wellness Guardrails** (`assembler._build_wellness_guardrails`): Персонализированные safety-правила — аллергии, Drug-Food взаимодействия, disease-specific макро-лимиты, интерпретация биомаркеров.
- **Layer 3 — Lifestyle Parameters** (`assembler._build_lifestyle`): Бюджет, гео-адаптация, голодание, стиль готовки, тренировки, предпочтения.
- **Маршрутизация**: `ArchetypePromptFactory.get_system_role(goal, gender, age, womens_health)` → возвращает собранный System Prompt + `archetype_code` для логирования (e.g. `DEFICIT_F_MATURE`).


**Realtime Service (общение в реальном времени)**
- WebSocket для семейной синхронизации
- Push-уведомления о приёмах пищи и витаминах
- Технологии: Node.js + WebSocket

**Auth & Status Service (статусы и идентификация)**
- Статусы: White / Black / Gold
- Интеграция с ЮKassa
- Технологии: Go или Node.js

**Push Service (уведомления)**
- Firebase Cloud Messaging (Android)
- Apple Push Notification Service (iOS)
- Расписание: напоминания о еде, витаминах, воде, тренировках

### Слой 4 — Хранилища данных
- PostgreSQL — содержит ТОЛЬКО публичные данные: 16 000+ материалов врачей, базу ингредиентов, справочники витаминов. Медкарты юзеров здесь НЕТ.
- Redis — кеш и временные данные (OTP-коды, сессии, rate limiting)
- Object Storage — фотографии блюд (Yandex Object Storage, S3-совместимый)
- pgvector — векторная база для knowledge_base (16 000+ экспертных материалов от 30+ практикующих врачей, транскрибированных и разбитых на чанки с embeddings)

## 6. Photo Analysis v1.1 — как работает

Пользователь делает фото блюда → приложение отправляет фото на сервер → сервер сохраняет фото в Object Storage → Gemini Vision определяет блюдо и ингредиенты → оценивает размер порции → ищет нутриенты в справочнике блюд и базе USDA → дополнительно ищет в базе знаний врачей персонализированные рекомендации по этому блюду для данного пользователя → возвращает результат.

Пример ответа сервера:
- Блюдо: Цезарь с курицей
- Уверенность: 87%
- Калории: 340, Белки: 28, Жиры: 18, Углеводы: 22, Клетчатка: 4
- Порция: 280 г

Если уверенность ниже 70% — приложение просит пользователя уточнить блюдо. Время ответа — менее 5 секунд. Доступно только на статусе Gold.

## 7. Принципы UI, Форматирования данных и Tone of Voice

### Tone of Voice (AI Personalities)
- **Общение с пользователем**: Базовый UI общается на «ты» в поддерживающем формате. Движок Характеров ИИ должен звучать как доброжелательный продукт Apple: чисто, просто и понятно.
- Доступно 4 архетипа (выбираются на этапе O-2.5):
  1. `classic` (Наставник): Уважительный, спокойный, заботливый помощник.
  2. `buddy` (Свой человек): Дружелюбный, теплый, общается на легком сленге.
  3. `strict` (Тренер): Ориентирован на дисциплину и мотивацию, но без агрессии.
  4. `sassy` (Дерзкий): Остроумный, мотивирует через легкий юмор.
- Система (в `assembler.py`) инжектит выбранный архетип (через `ai_personalities.json`) в системный промпт, чтобы на 100% персонализировать опыт и повысить виральность контента.
- **Мотивация**: Вместо сухих запретов и предупреждений используются `MotivatingTipCard` с эмпатичным объяснением процессов (например, объяснение влияния недосыпа на гормоны или влияния стресса на срывы).

### Input Sanitization & Форматирование данных
- Для обеспечения высокого качества данных, поступающих в AI-слой и бэкенд, в приложении внедрен слой глобальной очистки и стандартизации ручного ввода: `InputFormatter`.
- Он автоматически приводит текст пользователя к единому медицинскому стандарту перед сохранением в локальную БД: делает первую букву заглавной, заменяет сленговые термины и аббревиатуры на эталонные (например, "витамин ц" -> "Витамин C", "д3" -> "Витамин D3", "омега 3" -> "Омега-3", "б12" -> "Витамин B12").
- **Strict API Schema Binding**: Все отправляемые клиентом данные жестко привязываются к Pydantic-схемам бэкенда. Любые списочные значения обязаны передаваться как чистый `List<String>`.
- **Data Fallbacks (JSON Парсинг)**: Модели во Flutter (например, `MealItem.fromJson`) используют многоуровневые фолбэки для ключей (например, вес ингредиента парсится из `grams` ?? `quantity` ?? `amount`), а числовые поля безопасно приводятся через `(j['key'] as num?)?.toDouble()`.
- **Клетчатка (Fiber) как обязательный макрос**: Клетчатка (`fiber`) зафиксирована как обязательное поле наравне с БЖУ на всех слоях (БД, FastAPI Pydantic, Dart `MealItem`, UI-карточки). Значение `target_daily_fiber` рассчитывается строго на клиенте по рекомендациям WHO/AHA (с поправками на пол, возраст и цель).
- **UI State Safety (Dropdowns)**: Во избежание `StateError: No element` при изменении зависимых чекбоксов, все вызовы `.firstWhere()` в мапперах UI обязательно снабжаются фолбэком `orElse: () => list.first`.
- Слой работает непосредственно в момент ввода (в экранах онбординга O-5, O-6, O-10, O-13, O-15) перед передачей данных в репозиторий, гарантируя консистентность State Management'а.
- **Anchor Scroll (O-16 Summary)**: При нажатии «Изменить» на любой секции O-16, экран запоминает ID секции в статическое поле `_scrollToSection`. При возврате из редактирования, `initState()` выполняет `Scrollable.ensureVisible()` к соответствующему `GlobalKey`, после чего очищает поле. Все 15 секций покрыты маппингом `_routeToSection`. Ключи маппинга ОБЯЗАНЫ посимвольно совпадать с константами из `route_names.dart`.
- **UI Mapping Consistencies**: Все ключи, передаваемые из O-2 (Цели) и O-9 (Режим сна) обязательно должны иметь зеркальные ключи-переводы в O-16 `_goal` и `_sleepPat`. В том числе, ключи `age_adaptation` и `irregular`.
- **Universal AI Guardrails (assembler.py)**: Промпт-ассемблер использует универсальный цикл по `profile.allergies`, `profile.effective_restrictions` и `profile.disliked_foods` для генерации отдельных строк-запретов для КАЖДОГО элемента. Поверх универсального цикла действуют специализированные гарды для молочки, глютена, веганства.
- **Strict Culinary Slicing Rule**: При любом упоминании нарезки, шинковки или кусков (кубиками, соломкой, слайсами) ИИ-генератор **обязан** указывать примерный физический размер в мм или см (например, "нарежьте кубиками 1х1 см"). Это гарантирует предсказуемость рецепта и консистентность калорийности при приготовлении.

## 8. Health Connect интеграция

Приложение запрашивает доступ к Apple Health (iOS) / Google Health Connect (Android) и получает:
- Сон (время засыпания, пробуждения, фазы)
- Шаги за день
- Тренировки (тип, продолжительность, калории)
- Вес (если пользователь взвешивается на умные весах)
- Данные сохраняются локально на устройстве и синхронизируются с сервером через поля health_sleep, health_activity, health_workouts в таблице daily_logs.

### Логирование еды и напитков (AI Correction Loop)
- **Внеплановые перекусы (`extra_snacks`)**: Сохраняются локально. Включают `name`, `calories`, `macros`, `timestamp`.
- **Логи напитков и алкоголя (`beverages`)**: Сохраняются локально. Включают `name`, `volume`, `abv`, `estimated_kcal`, `timestamp`.
- **Key Synchronization**: Для гарантии того, что AI увидит актуальные данные, клиент сохраняет логи с суффиксом даты (`_YYYY-MM-DD`). Запрос на генерацию плана (`PlanGenerationUseCase`) автоматически считывает логи именно за текущий день и передает их в `extra_snacks` / `beverages` Pydantic-поля.
- **AI-Коррекция**: Если калорийность перекусов превышает 15% от дневной нормы, AI-ассистент в `assembler.py` получает инструкцию «мягкой компенсации»: добавление клетчатки, легкого белка и снижение ГИ в плане на следующий день.
- **Алкогольный калькулятор**: Автоматический расчет этанола (`Объем * ABV * 0.8 * 7`). При наличии алкоголя в логах, AI принудительно добавляет в план следующего дня продукты-гепатопротекторы и электролиты (рассол, калий, зелень).

## 8. Zero-Knowledge & Privacy Architecture

### Принципы
1. Сервер никогда не хранит health-данные пользователя (симптомы, заболевания, лекарства, анализы, рацион, фото еды, дневники, персональный wellness-план, raw onboarding health profile)
2. Профиль существует только на устройстве (Offline-Only для чувствительных данных)
3. Базовая авторизация анонимная (UUID hash). Опционально — привязка email для sync статуса
4. Статусы повышаются через IAP (iOS) или ЮKassa web (РФ). Backend — unified entitlement

### Что сервер МОЖЕТ хранить (minimal billing account)
- `billing_account_id`, `email_hash`, `encrypted_email`
- `app_profile_id`, `anonymous_uuid_hash`
- `entitlement_status`, `source` (app_store / web / trial / reviewer_demo / manual_support)
- `expires_at`, `apple_original_transaction_id`, `yookassa_payment_id`
- `web_subscription_id`, `consent_logs`, technical security logs

### Поток генерации плана (Stateless Backend)
1. Устройство формирует обезличенный JSON из локального профиля
2. JSON не содержит: имя, email, device_id, геолокацию
3. Сервер получает JSON → RAG → Gemini → возвращает план
4. Сервер немедленно удаляет входной JSON из памяти
5. В PostgreSQL не создаётся никакой записи с медицинскими данными

### Восстановление статуса при смене устройства
- **iOS**: Restore Purchases (StoreKit) — восстанавливает IAP-подписку
- **Web**: Пользователь входит по email (magic code) → синхронизирует web-статус
- Данные профиля (health) НЕ восстанавливаются с сервера. Онбординг заново (или из iCloud/Google Backup)

### Соответствие законодательству
- **GDPR (ЕС)**: не применяется к обезличенному health payload. Billing данные (email, платежи) обрабатываются по GDPR если пользователь из ЕС.
- **152-ФЗ (РФ)**: не применяется к обезличенному health payload, который не хранится и не позволяет идентифицировать пользователя. **Но**: email, billing account, платёжные записи, consent logs и web-подписка являются персональными данными и требуют соблюдения 152-ФЗ.
- **HIPAA (США)**: не применяется — сервис не является covered entity
- Медицинский дисклеймер: "ejeweeka не является медицинским устройством."

### Post-MVP: Group/Family Gold — E2EE синхронизация
> Перенесено в Post-MVP. При реализации:
> - Лидер группы генерирует локальный group_key (AES-256)
> - Данные шифруются локально, на сервер уходит encrypted_blob
> - Сервер ключа никогда не видит

### Smart Recipe Cache (Умное кеширование рецептов)
Хотя медицинские данные пользователей не сохраняются, приложение накапливает базу **обезличенных рецептов**. 
- При генерации плана сервер сначала ищет подходящие рецепты в PostgreSQL базе (по калориям, аллергиям и типу диеты).
- В запросе от устройства передается массив `exclude_recipes` (ID недавних блюд), чтобы блюда не повторялись чаще 2-3 раз в неделю.
- Если в базе недостаточно уникальных рецептов для статуса (например, нужно 3 варианта на выбор для Gold, а найден только 1), сервер вызывает LLM (Gemini) для генерации недостающих вариантов.
- Сгенерированные новые рецепты навсегда сохраняются в БД.
- **Imagen 4 Negative Prompting**: При генерации фотографий рецептов (через Imagen 4) аллергии и нелюбимые продукты (excluded_allergens) передаются как строгий negative prompt (`DO NOT show, depict, or include ANY...`), гарантируя отсутствие запрещенных ингредиентов даже в виде украшений.
- **Результат:** 95% запросов обрабатываются мгновенно из БД без вызова платного API нейросети, гарантируя разнообразие и выбор.

### Smart Workout Engine (Умная Маршрутизация Тренировок)
Приложение генерирует персонализированные планы тренировок без риска LLM-галлюцинаций (травмоопасности):
- LLM не генерирует упражнения. Вместо этого Python-скрипт (`workout_router.py`) действует как детерминированный маршрутизатор.
- База проверенных тренировок (`WorkoutCache`) содержит теги противопоказаний (`осевая_нагрузка`, `колени` и т.д.).
- Движок сопоставляет ограничения пользователя из `ProfileModel` (например, "Грыжа") с тегами тренировок и отбрасывает всё опасное.
- Применяется автоматическая периодизация (сплиты): если пользователь тренируется 3 раза в неделю, алгоритм ставит `FullBody` тренировки с отдыхом минимум 48 часов между ними.
- В UI (экраны `P-1` и `W-1`) план отображается в виде интерактивных чек-листов (по подходам) с таймерами отдыха и тактильной отдачей (HapticFeedback).

## 9. Статусы (MVP)

**MVP-статусы: white / black / gold.**

- **white** (бесплатный): план на 3 дня, 1 вариант блюда, без рецептов, базовый прогресс. Аллергии/заболевания/ограничения учитываются всегда.
- **black** (IAP: monthly/yearly; Web РФ: 490 ₽/мес): план на неделю, 2 варианта блюд, рецепты, витамины, совместимость лекарств, Health Connect, Умный отчёты, 5 тем.
- **gold** (IAP: monthly/yearly; Web РФ: 990 ₽/мес или 9 990 ₽/год): 3 варианта блюд, фото-анализ (5/день), тренировки с видео, 7 тем.
- **Триал**: первые 3 дня — полный gold-доступ. Стартует автоматически после завершения онбординга и генерации плана (при первом входе на H-1). Не активируется через O-17. После trial → white, если не оформлен black/gold.

> Примечание: в продукте используется термин «статусы» (Статус white, Статус black, Статус gold).

## 10. Инфраструктура

### 10a. MVP (0 → 50K пользователей)

**Принцип**: Managed-сервисы. Ноль DevOps. Максимум скорости запуска.

- Старт: **$0/мес**

- 50K юзеров: **$500-1500/мес**

### 10b. Scale (50K+ → 1,000,000 пользователей) — HighLoad

**Когда переезжать**: когда база активных пользователей превысит 50,000 и появятся жесткие требования по отказоустойчивости (SLA 99.99%).

**Где разворачиваем**: Cloud-провайдер с Managed Kubernetes (Yandex Cloud, AWS, GCP).

**Архитектура HighLoad (Microservices + Kubernetes)**:
- **API Gateway / Ingress**: NGINX Ingress с rate-limiting и WAF (защита от DDoS).
- **Auth Service**: 3+ подов (балансировка).
- **Core API**: Автоскейлинг (HPA) от 5 до 50+ подов в зависимости от нагрузки.
- **Recommendation Engine (Queue worker)**: Асинхронные воркеры на базе Celery/RabbitMQ или Kafka. Запросы к LLM не должны блокировать главный поток API.
- **Realtime Service**: 5+ подов за Load Balancer с настроенным Sticky Sessions.

**Базы данных (High Availability)**:
- **PostgreSQL HA Cluster**: Master-Slave репликация (1 Master для записи, 3+ Replicas для чтения). Обязательный шардинг логов, если они будут.
- **Redis Cluster**: Для сессий, rate limiting и кеширования частых ответов API.
- **CDN (Content Delivery Network)**: Раздача фотографий рецептов (Cloudflare, G-Core) по всему миру (Edge caching), чтобы снизить нагрузку на Object Storage.

**Мобильный клиент (Обновления)**:
- Использование Over-The-Air (OTA) обновлений (Capacitor Live Updates или CodePush), чтобы доставлять минорные фиксы UI миллиону пользователей в обход долгого ревью AppStore.

**Мониторинг и Observability**:
- Prometheus + Grafana (системные метрики, latency).
- ELK Stack (Elasticsearch, Logstash, Kibana) или Datadog для централизованного сбора логов всех микросервисов.
- Sentry (аллерты об ошибках в реальном времени с трейсингом).
